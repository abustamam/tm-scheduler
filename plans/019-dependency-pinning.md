## Why this matters

Nine dependencies — including the **production HTTP server runtime** — have no
version floor in the manifest:

- `"nitro": "npm:nitro-nightly@latest"` — the server that terminates every
  production request is an unversioned *nightly* (currently resolved to
  `3.0.1-20260626-071224-83acc441`, whose transitive request stack is
  RC/alpha: h3 2.0-rc, srvx 0.11, unstorage 2.0-alpha).
- Eight `@tanstack/*` packages pinned `"latest"`.

CI and the Dockerfile use `--frozen-lockfile`, so builds are reproducible
*until* anyone re-resolves: a local `bun update` (or any install that touches
resolution) silently jumps the router/start/query stack — or the nightly
server — across breaking versions with no diff in `package.json`. Pinning to
the currently-locked versions is behavior-preserving and makes the manifest
encode the supported range. While in the file, also move `drizzle-kit` to
`devDependencies` (the runtime never imports it — the bundled
`scripts/migrate.ts` uses `drizzle-orm/node-postgres/migrator` directly).

ranges — do not touch any other entry.

Context you should know:

- `bun install --frozen-lockfile` is used by CI (`.github/workflows/ci.yml`)
  and the Dockerfile build stage. Editing manifest ranges (and moving a dep
  between groups) changes what the lockfile must record, so `bun.lock` must be
  regenerated once (plain `bun install`) and committed together with
  `package.json`.
- The goal is a **no-op resolution**: after the edit, every resolved version
  in `bun.lock` should be exactly what it was. If any resolved version
  changes, something re-resolved — see STOP conditions.
- `extension/` has its own lockfile and no `latest` pins — out of scope.

## STOP conditions

Stop and report back if:

- `bun install` changes any RESOLVED version in `bun.lock` (not just ranges) —
  bun re-resolved something; do not proceed with a silent upgrade. Report
  which packages moved.
- Moving `drizzle-kit` to devDependencies breaks `bun run build` or any
  `db:*` script (it shouldn't — build-stage installs include devDeps — but if
  it does, that's a Dockerfile assumption worth reporting, not patching here).
- `bun install --frozen-lockfile` fails after re-lock.

## Maintenance notes

- **Upgrades are now deliberate**: bumping TanStack or Nitro means editing the
  manifest range and re-locking, reviewed like any change. Recommend replacing
  the nitro nightly with a stable `nitro@3.x` release when one exists —
  that's a real migration (test the vite plugin integration), tracked
  separately, not part of this pin.
- The TanStack packages cross-depend (react-start pins an exact react-router
  internally) — when upgrading, move them together.
- If Renovate/Dependabot is ever added, these explicit ranges are what make
  it useful; `"latest"` defeats it.
